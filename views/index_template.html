This project explores network data from Github. Github is web service that allows software developers to collaborate on and share open source software projects. It is focused on the git version control system: a hub for git repositories. Github has gradually added user interaction features that effectively make Github a social network. This is consistent with their tagline "Github: Social Coding". The entity-relationship diagram above shows all Github entities and the possible relationships amongst them. Our current analysis will focus on the user and repository (project) entities and the "starred" relationship between users and repositories. Starring is a method of a user indicating interest in a repository. This is analogous to a favorite or like in other social networks. Github maintains an API which makes the retrieval of these entities and relationships relatively straight forward.


This project will explore the relationship between users and repositories on Github. Particularly I am interested in exploring ways to visualize the distribution of different programming languages being used on Github and the user interactions of developers in the context of the languages they use.

Hypothesis

Developers who develop in certain languages only interact with other developers who focus on similar languages.

About Visual #1

Data

This graph makes use of the Github Public Timelime Event API to show starred repositories on Github, grouped by programming language. Each time a Github user stars a repository, it is captured in real-time. Each new star is essentially a new edge in our network data graph, so it is important to understand the general rate at which these edges are being created and some approximation of the distribution across languages. The chart is updated in real-time and plotted using 5-minute intervals. For smoothing, a 24 observation moving average is calculated and used for plotting. This results in a 2 hour (24 observations * 5 minute interval) window for the smoothing.

Horizon Chart

This graph makes use of a type of chart known as the Horizon Chart. This paper describes the Horizon Chart and cites the work of Edward Tufte (specifically small multiples and the data/ink ratio) and Cynthia Brewer (known for her work on color theory in cartography). The Horizon Chart reduces the use of vertical space without losing resolution by overplotting larger values in successively darker colors.

Analysis

It is important to note that values for each language are plotted on an independently scaled y-axis and therefore values should not be compared across languages, rather relative values should be compared across languages. Changes in the number of repositories starred are most easily identified by changes in the coloring and height of the horizon line. It is interesting to note that there appears to be a spike across several languages at approximately 2am each weekday. It would be interesting to gather data on which specific projects were being starred at certain time intervals and the distribution of unique repos being starred. Are the spikes due to a popular news/blog post generating buzz for a specific project? This data would most likely be more useful if plotted over a longer time interval, however due to data collection complications this was not explored for this visual.

How Created

This visual was created using the following open-source libraries:

  • Cube - a system for collecting timestamped events and driving metrics
  • Cubism.js - a JavaScript library for time series visualization
  • D3 - data driven documents.
A server process is constantly running to consume the Github Public Timeline Event API. Each event is inserted into mongoDB on the server. When the webpage is loaded a WebSocket connection is established between the client and server to populate the graph. The WebSocket connection is maintained to allow for continuous real-time updates of the graph (in this case in 5-minute intervals).

Dataset

The visual displayed above uses data from the traversal of my Github social network (friends and followers). A node is created for each friend or follower (user nodes). For each user node, additional nodes are created for each starred repository, with an edge from the repository node to the user node of the user who starred that repository. Repository nodes with more than one edge show that multiple users have starred the same repository. The Alternate Visual linked to above uses similar data, but rather than a traversal of my Github social network, relies on data from the top Github contributors.

About Visual

This visual was creating using D3 Javascript library. It uses a force directed layout to position each node. The forces being simulated are a repulsive charge (all nodes) and an attractive force between connected nodes. This results in nodes with similar edges being grouped together on the graph and nodes with dissimilar connections being pused away to the edges of the visual. reated with D3 force directed layout. Additional information about each node is available by clicking on the node.

Analysis

This graphic shows several repository nodes focused around individual user nodes: the repositories that particular user has starred. The interesting nodes are those starred by multiple users. These nodes can be identified as those with multiple edges. The force directed layout has the effect of pulling these nodes out of the cluster associated with a single user and into a cluster of nodes shared by two or more users.

Analysis

An interesting characteristic that appears in both visuals presented above is the apparent distinction of two classes of languages. In the center of the graph are common language nodes (Java, C, Python, Javascript, etc) that more users appear to be interested in. On the outer regions of the graph are more esoteric languages (LiveScript, F#, TypeScript, OCaml, etc) that are used by one or only a few users. This is Appear to be two classes of languages: common language with edges to many user nodes, and many uncommon langues with few edges.

Analysis And Conclusions

Power Law Analysis

A power law function expresses the value of a variable in terms of the power (exponent) of another variable. In this case we explore the power law relationship of the distribution of degree of the nodes in our graph. In other words, we look for evidence that the frequency of the number of edges per node follows a power law.

Conclusions

In terms of the power law distribution, this analysis shows that many repositories are starred by one user (in our sample) and few repositories are starred by multiple users. This makes sense in the context of our data, but it would also be interesting to include all stars per repository and not just the stars of users in our sample. This would be a more consistent comparison with the standard popularity power law function.

Conclusions

Most of the evidence presented in the visualizations appears to be inconsistent with our hypothesis. Rather than ego-type network patterns, with many developers focused around one or two languages, we observed many developers using many different languages. Further analysis could be done to examine the hypothesis:

  • Edges could be weighted according to number of contributions in a specific language
  • Data on more users could be collected to extend the dataset
  • Data on more user interactions (contributions, following, pull requests, forking, branching, etc ) all with different edge weights according to their relative significance would be particularly imformative
